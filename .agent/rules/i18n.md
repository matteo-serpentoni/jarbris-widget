# Internationalization (i18n) Rules ‚Äî jarbris-widget

These rules define how multi-language support is handled in the widget.
They are specific to the widget's zero-dependency, embedded-third-party architecture.

## 1. Translation Infrastructure

- **No external i18n library.** The system is a hand-rolled singleton in `src/i18n/index.js`.
- **Locale bundles**: stored as nested JSON files in `src/i18n/locales/` (`en.json`, `it.json`, `es.json`, `fr.json`, `de.json`, `pt.json`).
- **Key format**: dot-notated namespaces (e.g., `checkout.preparing`, `product.sold_out`).
- **Fallback language**: `en`. All keys MUST exist in `en.json` first.

## 2. Translation Function

```js
import { useI18n } from '../../hooks/useI18n';

// Inside a component:
const t = useI18n(); // returns the t() function directly ‚Äî NOT an object
t('namespace.key');
t('product.only_left', { count: 3 }); // interpolation via {{count}}
```

> **CRITICAL**: `useI18n()` returns `t` directly. NEVER destructure it as `const { t } = useI18n()`.

Outside React (utils, non-component code), import `t` directly from the singleton:

```js
import { t } from '../../i18n';
```

## 3. User-Facing Strings

- **Never hardcode** user-visible text in JSX or JS logic. Always use `t('key')`.
- **Scope**: only text visible to the shopper (labels, button text, status messages, ARIA labels, placeholder text, error messages).
- **Exclusions**: do NOT translate internal log messages, CSS class names, API signal strings, or developer-facing values.

## 4. Language Resolution

Priority order (handled automatically by the singleton):

1. `bootData.lng` from the API `/boot` response (highest priority ‚Äî set via `setLng()`)
2. `navigator.language` (browser preference ‚Äî detected at module load)
3. `'en'` ‚Äî default fallback

> The widget adapts to the **visitor's browser language**, overridden by the API's language signal.
> `setLng()` normalizes locale codes: `fr-FR` ‚Üí `fr`, `pt-BR` ‚Üí `pt`.

## 5. Adding New Strings

When adding new user-facing text in ANY component or hook:

1. Add the key to `en.json` first (source of truth)
2. Add the translation to ALL other locale files: `it.json`, `es.json`, `fr.json`, `de.json`, `pt.json`
3. Use `t('namespace.key')` in the code ‚Äî never raw strings
4. Group keys by domain namespace (e.g., `checkout.*`, `product.*`, `feedback.*`, `ui.*`)

## 6. Namespace Reference

| Namespace      | Contents                                          |
| -------------- | ------------------------------------------------- |
| `chat.*`       | Chat shell labels, placeholders, legal text       |
| `session.*`    | Session expiry messages                           |
| `errors.*`     | Generic error messages                            |
| `feedback.*`   | Star rating, helpful/not helpful                  |
| `profile.*`    | Profile editor & privacy                          |
| `profile_card.*` | Profile invitation card                         |
| `ui.*`         | Generic UI chrome (buttons, fallbacks, toasts)    |
| `checkout.*`   | All checkout flow states                          |
| `connection.*` | Socket status labels (ONLINE, OFFLINE, etc.)      |
| `promo.*`      | Promo card details                                |
| `product.*`    | Product cards, sheets, stock, cart                |

## 7. SSR / Non-Browser Safety

- The singleton guards against missing `navigator` at module load.
- `detectLang()` falls back to `'en'` if `typeof navigator === 'undefined'`.
- This ensures the singleton is safe in Jest (Node) and any non-browser context.
